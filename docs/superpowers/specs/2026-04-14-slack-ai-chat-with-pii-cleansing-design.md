## Slack AI Chat + 自動クレンジング（PII/機密）プロトタイプ 設計（Design Spec）

- **作成日**: 2026-04-14
- **目的**: Slack DM から自然に投げられる AI チャットを提供し、ユーザーが意識しなくても **個人情報/機密情報がクレンジングされた状態で外部LLM（Gemini API）に送られる**ことを、プロトタイプとして検証・デモ可能にする。
- **想定ユーザー**: 社内メンバー（将来は会社Slack導入を想定、まずは個人Slackワークスペースで検証）

---

## 1. ゴール / 非ゴール

### ゴール（MVPで必ず満たす）

- **Slack DM** でボットに投稿すると、AI応答が返る
- 外部LLMへ送信する前に、入力文が **自動クレンジング（マスキング）**される
- **DMの新規投稿（親メッセージ）= 新規セッション**として扱う
- **スレッド返信 = 同一セッション**として会話文脈を継続する
- 何をマスクしたかが **Slack上で可視化**される（少なくともスレッド最初の bot 返信）
- マスクした情報を **セッション内で復元**して、自然な応答にできる（復元範囲はポリシーで制御）
- データ所在地要件のため、状態管理ストアを **日本リージョン（Tokyo）**に置ける構成とする

### 非ゴール（このプロトタイプでは保証しない）

- DLPを完全に満たすことの厳密保証（取りこぼしゼロ）
- 会社Slack本番導入（SSO/SCIM/監査要件フル対応）
- 添付ファイル（画像/ドキュメント）やリンク先本文のクレンジング
- “疑わしい場合のみローカルLLMで判定”の Step2 実装（設計には拡張として記載）

---

## 2. ユースケース / UX

### 2.1 会話開始とセッション

- ユーザーがボットに **DMで新規投稿（親）**する
  - ボットは **その親メッセージのスレッド**を作成し、以降はスレッド内で返信する（セッションの場）
- ユーザーが **スレッドに返信**する
  - ボットは **同一セッション**として、会話履歴を考慮して応答する

### 2.2 返信の構成（推奨）

スレッド最初のボット返信は、以下を含む。

- **(A) マスク結果サマリ**（必須）
  - 例: `Masked: EMAIL x1, PHONE x2, PERSON x1`
  - 例: `Policy: usefulness-first (B)`
- **(B) AIの回答**（復元後の文面）

マスク結果の “プレビュー” をどこまで出すかは要注意（ログ/Slack表示に生値を出さない方針）。

---

## 3. 全体アーキテクチャ（案1: 外部送信前にクレンジング完了）

### 3.1 構成要素

- **Slack App / Bot**
  - DMイベント、スレッド返信イベントを受信
- **アプリ本体（HTTPサーバ）**
  - Slack Events 受信、クレンジング、Gemini API呼び出し、Slack返信
  - デプロイ先は **Cloud Run を第一候補**（最速でデモでき、将来GCEへ移植しやすい）
- **クレンジング基盤（Step1）**
  - **Microsoft Presidio**（Analyzer/Anonymizer）を利用
- **状態管理ストア（日本リージョン）**
  - **Firestore（Native mode, `asia-northeast1` Tokyo）**を想定
  - セッション単位の `mask_map` とサニタイズ済み履歴を TTL 付きで保持
- **外部LLM**
  - **Gemini API**

### 3.2 データフロー（テキストのみ）

1. Slack からイベント受信（DM親 or スレッド返信）
2. イベント正規化（`user_id`, `channel_id`, `text`, `ts`, `thread_ts` 等）
3. **PresidioでPII検出→マスク**（`sanitized_text` と `mask_map` を生成）
4. セッション状態を読み出し、**サニタイズ済み履歴**として Gemini に投入するプロンプトを組み立て
5. Gemini API 呼び出し（外部送信は **サニタイズ済み**のみ）
6. Gemini の応答テキストに対して、`mask_map` を使い **復元**（ポリシーに従う）
7. Slack スレッドへ返信（最初の返信はマスク結果サマリ付き）
8. セッション状態を Firestore に保存（TTL対象）

### 3.3 Slack 連携方式（プロトタイプ）

- **Slack Events API + 公開HTTPエンドポイント** を前提（Cloud Run想定）
- 検証/セキュリティ観点で以下を実施する
  - **Signing Secret** によるリクエスト検証
  - リプレイ対策（Slackの `X-Slack-Request-Timestamp` 等の検証）

## 4. クレンジング設計（Presidio）

### 4.1 方針（B: 業務有用性優先）

- できる限り意味を保ちつつマスクする（過剰マスクは避けたい）
- ただし **外部LLMへ生値を出さない**が最優先（クレンジングは外部送信前に必須）

### 4.2 マスクトークン形式

- 種別付き・連番付きのトークンに置換する
  - 例: `<EMAIL_1>`, `<PHONE_1>`, `<PERSON_1>`, `<LOCATION_1>`
- `mask_map` は `{ "<EMAIL_1>": "alice@example.com", ... }` のように保持

### 4.3 Presidio 設定（例）

- **Analyzer**: まずは Presidio 既定の検出器（EMAIL_ADDRESS, PHONE_NUMBER, CREDIT_CARD, PERSON, LOCATION など）
- **カスタム**: 会社導入時を見据え、将来以下を追加可能
  - 社員番号/顧客IDなどの正規表現
  - 会社固有の辞書（プロジェクト名や顧客名など“機密扱い”語彙）

### 4.4 Step2（拡張案）: 疑わしい場合のみローカルLLM判定

- Presidio の “低信頼スコア” や曖昧ケースをフックにしてローカルLLMで再判定
- **プロトタイプでは未実装**（設計に残し、将来の選択肢として合意）

---

## 5. 復元（De-mask）設計

### 5.1 復元の目的

- Slack への返信は自然な文章にしたい（固有名詞が `<PERSON_1>` のままだと読みにくい）

### 5.2 復元の制約（重要）

- **復元はセッション内（= 同一スレッド）に限定**
- Firestore に保存する `mask_map` は TTL 付きで短期保持（例: 24h）
- 監査ログ/アプリログに **生値（original）を出さない**

### 5.3 復元ポリシー（案）

初期案はシンプルに “全部復元” ではなく、以下のような段階導入を想定する。

- **ポリシー P0（プロトタイプ推奨）**: PERSON/EMAIL/PHONE 等は復元するが、CREDIT_CARD 等は復元しない（マスク維持）
- **ポリシー P1（将来）**: ユーザーがスレッド内コマンドで切替（例: `mode: strict/useful`）

---

## 6. セッション状態管理（Firestore, Tokyo）

### 6.1 セッションキー

- `session_id = "<channel_id>:<thread_ts>"`

### 6.2 保存するデータ（例）

- `mask_map`: token → original（復元用、TTL対象、ログには出さない）
- `history`: サニタイズ済みの会話履歴（直近N件）
- `created_at`, `updated_at`, `ttl_at`
- `mask_summary`: 種別別件数（Slack表示/監査用に利用）

### 6.3 TTL / データ削除

- セッション状態は短期保持（例: 24h）を基本とし、Firestore の TTL 機能で自動削除
- 会社導入時は、要件に応じて保持期間・削除要件（Right to be forgotten等）を再検討
- Firestore の **ロケーションは後から変更できない**ため、プロジェクト作成時に `asia-northeast1`（Tokyo）を選ぶ前提で進める

---

## 7. セキュリティ / コンプライアンス観点（プロトタイプの説明用）

### 7.1 外部送信

- 外部LLM（Gemini API）へ送信するのは **サニタイズ済みテキストのみ**
- Firestore（Tokyo）に保持する `mask_map` は復元用で、外部送信しない

### 7.2 ログ

- ログに残すのは原則 **メタデータのみ**
  - 例: `session_id`, `mask_summary`, `latency`, `error`, `model`
- Slack上のマスクサマリ表示も **生値は載せない**

### 7.3 シークレット

- Slack / Gemini のトークンは Secret Manager 等で管理する（コード/ログに含めない）

---

## 8. 可観測性（最低限）

- **メトリクス**
  - 1. リクエスト数、2) latency（Slack受信→返信まで）、3) Gemini呼び出し時間、4) エラー率
- **クレンジング品質の観測**
  - `mask_summary` の分布（種別・件数）
  - “ユーザーがマスク誤りを報告”できるフック（将来）

---

## 9. オープンクエスチョン（要レビュー）

- **復元ポリシー**: どの種別を復元するか（P0の範囲）
- **Slack表示**: マスク結果の表示形式（サマリのみ vs 置換後テキスト一部）
- **誤検出対応**: “これは機密ではない” / “これは機密だ” のフィードバックループをどう入れるか
- **ローカルLLM Step2**: 実装する場合の運用（GCE常駐、モデル、コスト、性能）
- **添付対応**: ファイル/画像/リンク先の扱い

---

## 10. 段階導入ロードマップ（案）

### Phase 0: デモ最速（本 spec の範囲）

- Slack DM + スレッドセッション
- Presidio Step1 でマスク
- Firestore（Tokyo）に `mask_map`/履歴保持（TTL）
- Gemini API で回答
- マスクサマリをスレッド最初の返信に必ず付与
- 復元（P0）で自然な回答に

### Phase 1: 品質・運用強化

- “疑わしい場合のみ”ローカルLLM Step2 判定
- ユーザーがスレッド内でモード切替（strict/useful）
- 誤検出のフィードバック導線

### Phase 2: 会社Slack導入準備

- 監査/アクセス制御、運用Runbook、権限設計、SLO
- データ保持・削除ポリシー
